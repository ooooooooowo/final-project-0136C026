
# final-project-0136C026

## 心得

- **背景**: 我以 `fff` 的財報抓取與評估引擎為基礎，目標是把雜亂的會計資料轉成可操作的結構化投資判斷。
- **設計取向**: 優先「可用性」與「可解釋性」：面對不同公司與欄位命名，採欄位備援與安全除法，並以杜邦、ROIC、FCF 等直觀指標做報告輸出。
- **技術抉擇**: 使用 pivot/merge_asof 處理時序對齊、`np.clip` 限制稅率、對投資資本與市值作合理保護（最小值 1），這些都是為了避免極端值造成錯誤結論。
- **主要挑戰**: 第三方資料欄位差異與資料缺漏是核心痛點；為此採用了多重備援欄位與寬容的缺值處理，但這也帶來指標意義可能被弱化的風險。
- **實務心得**: 實務上「穩健勝過完美」——能跑通、並產出可解釋的診斷，比追求完美一致性的數值更有價值；報告語句化（可讀性）對決策者很重要。
- **改進方向**: 建議加單元測試與 mock 資料、把資料載入與分析解耦、改用 logging 與明確例外類型，並將診斷規則抽成可配置的規則檔以利調校。

在實作這套財報分析引擎的過程中，我深刻體會到資料工程與財務知識的交叉重要性。程式不只是把公式寫出來，更多時間花在處理「資料不純粹」的現實：欄位錯配、缺值、單位不一致、以及報表口徑的差異。這些看似枯燥的邊界情況，實際上決定了整個系統能否在真實世界穩定運作。

一開始我傾向於把指標寫得「嚴格且精準」，但很快發現這會讓程式在面對少量差異時就崩潰。於是我採用了更務實的策略：多重備援欄位、保守的安全除法、以及明確的資料完整性檢查。這個折衷帶來的好處是系統能夠對大多數公司「跑得通且產出可讀的報告」，但代價是某些指標在語義上變得不那麼純粹，因此輸出前常需要人工或二次檢核以確認合理性。

在評估規則方面，我採取可解釋性為第一原則：將 ROIC、杜邦分解、FCF Yield、CapEx/D&A 等指標，搭配簡單的閾值與文字化診斷，讓報告更容易被非工程背景的投資人理解。這種做法雖然不保證最佳化投資績效，但在真實決策流程中，透明與可溝通性往往比黑箱準確度更有用。

技術上，我也體會到「測試」與「可重現性」的重要性。對這類資料驅動系統，建議的改進路徑是：將資料載入層抽離（便於 mock）、建立涵蓋缺欄、極端值、時序錯位等情境的 unit tests、以及在 CI 中加入實際小規模資料的端到端測試。如此一來，每次改動都能快速驗證是否破壞處理邏輯。

展望未來，若要把此系統推向更穩定的產品化路徑，我會優先做到：設定化（把閾值與規則放進設定檔）、加強觀察性（`logging`、metrics、錯誤分級）、實作快取與重試機制、並為 API 呼叫與資料處理加入監控與告警。對於使用者介面，可考慮把文字化報告輸出為 Markdown 或 PDF，並加上重點高亮，讓非技術使用者更容易採納。

最後，我認為這個專案的核心價值在於把複雜的財務數字「翻譯」成可操作的結論與風險提示。程式碼本身只是工具，真正的價值在於結合財務判斷、工程實作與良好的使用者體驗，讓投資決策既科學又可理解。

## 遇到的問題

- 資料欄位不一致：不同公司或不同時間點的 FinMind 回傳欄位命名可能不同（例如 `Asset` vs `TotalAssets`、`OperatingCosts` vs `CostOfGoodsSold`），導致 KeyError 或計算錯誤。
- 資料缺漏與不完整：有些公司未提供完整三表或現金流資料，必須設計備援邏輯或在上游拒絕分析。
- 時序對齊問題：財報日期、股價日期與報表公告日非同步，若合併策略不當會造成錯誤的指標對應。
- 單位與口徑差異：金額單位（千／萬／元）、合併報表或個體報表差異，若未統一會導致錯誤的比率與市值計算。
- 極端值與分母為零：營收/權益為零或為負會造成除法爆炸，需採用安全除法與下界保護（例如最小為 1）。
- CAPEX 與現金流口徑不一：不同公司在 cash flow 的分類上有差異，必須對 capex 做多欄位備援，且理解代理欄位的語義偏差。
- API 與網路穩定性：FinMind 的可用性、速率限制或網路錯誤會導致流程中斷，需重試、退避與快取策略。
- 效能與延遲：抓大量歷史財報與股價時，請求耗時與記憶體使用量需考量，最好加批次或分段抓取。
- 測試與 mock 困難：真實資料包含各種雜訊，建立能涵蓋多情境的 mock 資料需要花心力。
- 使用者輸入驗證：錯誤的股票代碼或日期格式會導致空結果或例外，需友善提示與驗證。

最後，我認為這個專案的核心價值在於把複雜的財務數字「翻譯」成可操作的結論與風險提示。程式碼本身只是工具，真正的價值在於結合財務判斷、工程實作與良好的使用者體驗，讓投資決策既科學又可理解。
